# Integration Analysis Pipeline

## 1. Introduction and Overview

### 1.1 Purpose

This document describes the integration analysis pipeline. The pipeline consumes
unit inventory files and produces project-level integration analysis artifacts.

Unit analysis identifies integration points and callable-local EI paths that
reach those points. Integration analysis composes those unit-scoped facts across
inventories to reason about seams between units, project-level call structure,
reachable flows, graph findings, and integration test planning.

**Key Distinction:**

- **Unit analysis** records callable-owned execution items, unit-scoped
  integration points, and paths to those integration points.
- **Integration analysis** composes unit inventories to analyze interactions
  across callables and units.

### 1.2 Goals

The integration analysis pipeline should:

- collect integration candidates from unit inventories
- normalize and classify seams across units
- build a project-level callable and integration graph
- identify seam test obligations from execution paths to integration points
- use feature flow markers to trace meaningful feature paths across units
- model all feature branching points, convergence points, and terminal points
- detect graph findings that may require review or targeted probe tests
- provide evidence for integration test planning
- preserve provenance back to unit inventories, EIs, source expressions, and
  execution paths

### 1.3 Input

The primary inputs are:

- unit inventory files produced by the unit analysis pipeline
- optional project discovery or inventory indexes
- optional marker/decorator metadata captured during unit analysis
- optional knowledge sources for standard library, external library, and
  boundary classification

### 1.4 Output

The pipeline may produce multiple artifacts:

1. **Integration candidate collection**: normalized candidates collected from
   unit inventories
2. **Classified seams**: accepted integration points classified by target,
   boundary, role, and confidence
3. **Integration graph**: graph representation of callable and integration
   relationships
4. **Seam test specifications**: focused test obligations for each reachable
   seam
5. **Feature flow specifications**: marker-guided feature flows with entry
   points, waypoints, branches, convergence points, and terminal points
6. **Graph analysis report**: findings, review signals, validation results, and
   structural diagnostics

## 2. Integration Test Specification Types

Integration testing is divided into two main specification types: seam test
specifications and feature flow test specifications:

1. Seam specs are the lightest integration testing layer. They verify individual
cross-unit or boundary interactions using the EI paths that reach each seam.

2. Feature flow specs are higher-level. They use explicit feature flow markers
to trace meaningful behavior through multiple units. A feature flow may include
many seams, but it is not generated by blindly expanding every possible graph
path. It is guided by marker metadata that identifies feature starts, waypoints,
branches, convergence points, and terminal outcomes.

## 3. Stage-by-Stage Procedure

### Stage 1: Build Inventory-Based Call Graph

Consumes unit inventories and builds the execution-instance-level graph. This
stage indexes callable contexts, EI ownership, callable fully qualified names,
local callable names, contract implementations, and relevant type metadata. It
adds within-callable flow edges, call edges, return edges, and collapsed
operation nodes where appropriate.

This stage also verifies that execution paths recorded in the unit inventories
are realizable in the assembled graph.

### Stage 2: Generate Integration Seam Test Specifications

Generates focused test specifications for seams found in the graph. Each spec
describes one source-to-target interaction, the representative execution paths
that reach it, the seam being tested, and the fixture or mock requirements
needed to isolate that seam.

### Stage 3: Organize Seam Test Specifications

Groups seam specs into files or test-generation batches. The default
organization is by a source-unit to target-unit pair. This keeps test obligation
generation separate from packaging decisions.

### Stage 4: Trace Marked Feature Flows

Uses feature flow markers to identify meaningful feature paths across the graph,
including feature starts, waypoints, branches, convergence points, and terminal
points.

### Stage 5: Generate Feature Flow Test Specifications

Generates higher-level integration test specifications from traced feature flows.

### Stage 6: Organize Feature Flow Test Specifications

Groups feature flow specs into test-generation batches according to feature,
entry point, flow family, or another stable organization rule.
