# Integration Analysis Pipeline

The PyBastion integration analysis pipeline consumes unit inventories and
produces integration test specifications.

The unit pipeline is responsible for analyzing individual source units. It
identifies callables, execution items, integration candidates, bindings, type
context, and paths to integration points. The integration pipeline starts after
that. It uses those unit-scoped facts to build a project-level view of how one
unit reaches another.

The goal is not to generate finished tests by itself. The goal is to make
integration seams explicit enough that a developer or coding agent can reason
about them without having to rediscover the project structure from raw source
code.

## Overview

Unit inventories describe what happens inside each analyzed source unit. That
is useful, but integration testing usually needs a project-level view.

The integration pipeline answers questions such as:

- Which source units call into which target units?
- Which execution items participate in those calls?
- Which callable-local paths can reach each integration point?
- Which seams are worth turning into integration test specifications?
- Which smaller specification files should be handed to a developer or agent?

The pipeline does this without pretending to be a full runtime interpreter. It
works from the structural and behavioral facts emitted by unit analysis.

That distinction matters. The integration pipeline is not supposed to guess
everything from source again. It should consume the inventory, preserve the
facts that are already known, derive project-level relationships, and keep
uncertainty visible when resolution is incomplete.

## Inputs

The primary input is the unit inventory generated by `pybastion-unit`.

By default, the integration pipeline reads inventories from:

```text
dist/pybastion/inventory/
```

Those inventories are expected to contain enough information to identify:

- source units
- callable entries
- execution items
- integration candidates
- resolved targets, when available
- execution paths to integration points
- supporting type and binding context

The integration pipeline should not require the original source files for its
core graph and specification work. Source may still be useful for diagnostics,
display, or future enrichment, but the inventory is the authoritative input for
this stage of analysis.

## Pipeline Stages

The current integration pipeline has three main stages:

1. Build the execution item call graph.
2. Generate integration test specifications.
3. Split specifications into smaller files.

The graph checker can optionally run after Stage 1.

### Stage 1: Build the EI Call Graph

Stage 1 builds a project-level graph from the unit inventories.

The graph is centered on execution items, not only on callables. This matters
because a callable-level graph can say that one function calls another, but it
does not explain where that call occurs, which path reaches it, or which
execution item owns the integration point.

The Stage 1 graph should preserve relationships such as:

- source unit
- source callable
- source execution item
- integration target
- target unit, when resolved
- target callable, when resolved
- paths that reach the integration point
- metadata needed by later stages

The graph does not need to be beautiful. It needs to be useful, inspectable,
and faithful to the inventories that produced it.

By default, Stage 1 writes:

```text
dist/pybastion/integration-output/stage1-ei-cfg.pkl
```

The pickle format is an implementation choice. It is convenient for the current
Python pipeline, but it should not be treated as the long-term public artifact
format unless that becomes an explicit decision.

#### Flattened Flow and Loop Representation

The EI call graph is built from unit inventory facts, not directly from raw
source code.

This means the graph inherits the unit pipeline's flattened flow model. Loops
are not expanded into unbounded runtime cycles. They are represented through
the finite execution item paths emitted by unit analysis, such as zero
iterations, one-or-more iterations, and disruptive paths.

This is intentional. The integration pipeline is trying to identify reachable
integration seams and produce useful test specifications. For that purpose, an
infinite or runtime-style loop graph is usually less useful than a finite,
inspectable set of representative paths.

For example, if a loop contains a call to another project unit, the integration
pipeline does not need to model every possible iteration count. It needs to
know whether the integration can be skipped, whether it can be reached, and
which execution path reaches the integration EI.

The graph should preserve those collapsed relationships. It should not restore
a source loop into a cyclic CFG unless a future graph mode explicitly chooses
to do that.

### Optional Graph Checker

The graph checker validates that the Stage 1 graph lines up with the source
unit inventories.

It is not a test generator. It is a sanity check for the analysis pipeline.

The checker should help detect problems such as:

- graph nodes that do not correspond to inventory facts
- missing callable or execution item references
- broken source-to-target relationships
- unexpected graph structure
- inventory facts that were not represented correctly in the graph

By default, the checker writes:

```text
dist/pybastion/integration-output/inventory-graph-report.yaml
```

The checker is optional because it is diagnostic. It is useful while developing
the analyzer, validating changes, or investigating strange output. It does not
need to run every time if the user only wants fresh integration specs.

### Stage 2: Generate Integration Test Specifications

Stage 2 consumes the EI call graph and generates integration test
specifications.

A specification is not a finished test file. It is a structured description of
a testable integration seam. It should give a developer or coding agent enough
context to decide what fixture, mock, fake, setup, input, action, and assertion
might be needed.

A generated integration spec should answer questions like:

- What source unit owns the integration?
- What target unit or callable is reached?
- What execution item contains the integration point?
- What path or paths can reach the seam?
- What behavior should be exercised?
- What test double or fixture might be needed?
- What uncertainty or missing context remains?

Stage 2 should avoid turning every syntactic call into a useful integration
test obligation. The point is to produce meaningful seam specifications, not a
pile of noisy call records.

By default, Stage 2 writes:

```text
dist/pybastion/integration-output/stage2-integration-test-specs.yaml
```

### Stage 3: Split Specifications

Stage 3 splits the Stage 2 output into smaller files.

The split output is grouped by source unit and target unit pair. This keeps the
specifications smaller and more practical for review or agent use.

By default, Stage 3 writes files under:

```text
dist/pybastion/integration-output/specs/
```

This split is intentionally practical. Large monolithic spec files are harder
to inspect and easier to misuse. Smaller files make it easier to hand a focused
task to a developer or an agent:

> Here is the source unit. Here is the target unit. Here are the seams between
> them. Generate or review tests for this relationship.

## Stage 4: Feature Flow Analysis

Stage 4 turns feature-flow analysis markers into testable integration flow
cases. Earlier stages identify executable items, outcomes, integration points,
and graph reachability. This stage uses that information to trace how a marked
feature moves through the project, including branch-specific behavior,
convergence, conditional completion paths, and shared downstream segments.

The goal is not only to find a path from the start of a feature to its end. The
goal is to identify the distinct feature cases, resulting from branch-specific
execution paths, that should become integration test obligations.

### Feature Branch Expansion and Flow Case Assembly

Feature-flow tracing is not a single shortest-path problem. A feature may contain
multiple marked branch points, and each branch point can expand the number of
testable feature-flow cases.

The tracer treats `FeatureStart` as the beginning of the feature's root `main`
case. Feature branches then create branch-specific cases. When branches converge,
the execution path may share a downstream segment, but the distinct feature cases
are not erased. Convergence affects path reuse and segment assembly, not test-case
identity.

For example:

```text
main
  -> branch group: a | b
  -> converge
  -> branch group: c | d
  -> converge
  -> FeatureEnd
```

This produces four testable feature-flow cases:

```text
main -> a -> c -> end
main -> a -> d -> end
main -> b -> c -> end
main -> b -> d -> end
```

The number of final feature-flow cases is driven by the branch frontier
encountered during traversal:

- Each branch group multiplies the active cases that reach it.
- Nested branch groups extend the current branch lineage.
- A nested branch group multiplies only the cases in its own lineage, not
  unrelated branches that do not reach it.
- Convergence joins the execution location but preserves the case identity that
  reached the convergence point.
- Converged branches share a common downstream segment.

To support this, marker-aware feature tracing should operate in three conceptual
passes:

1. Build a feature marker inventory from unit inventories.
2. Use `FeatureBranch`, `FeatureConverge`, `FeatureEnd`, and
   `FeatureEndConditional` markers to derive feature branch topology and
   testable flow cases.
3. Traverse the EI graph using branch-aware state, producing reusable path
   segments and then assembling full feature-flow cases from those segments.

Each segment records only its local EI path, but also carries its absolute branch
lineage. Full feature-flow paths are assembled later by combining the appropriate
main, branch, convergence, shared-tail, and end segments.

`FeatureConverge` does not reduce the number of test cases. It identifies where
separate branch paths can resume shared traversal. `FeatureEndConditional` marks
a conditional test obligation and may terminate one branch case while other cases
continue to a normal `FeatureEnd`.

## Outputs

A typical integration analysis run writes:

```text
dist/pybastion/integration-output/
  stage1-ei-cfg.pkl
  inventory-graph-report.yaml
  stage2-integration-test-specs.yaml
  specs/
```

The important outputs are:

- `stage1-ei-cfg.pkl`: the generated execution item call graph
- `inventory-graph-report.yaml`: optional graph checker report
- `stage2-integration-test-specs.yaml`: generated integration specs
- `specs/`: split specs grouped for focused review or downstream use

The exact artifact formats may change while the project stabilizes. The
important contract is the information being preserved: units, callables,
execution items, integration seams, paths, and test-relevant context.

## What the Specs Are For

The integration specs are meant to support test generation, not replace all
judgment.

They are useful because they gather the important context before a human or
agent starts writing tests. Instead of asking an agent to inspect an entire
project and infer everything from source, PyBastion can provide a structured
view of the seam being tested.

A good spec should help answer:

- What is the integration relationship?
- What code path reaches it?
- What setup is probably required?
- What target behavior needs to be controlled or observed?
- What output, state change, exception, or interaction matters?
- What assumptions should be checked before trusting the generated test?

This is why the specs are intentionally inspectable. They should make the
reasoning visible enough that a developer can reject, correct, or improve the
test generation step.

## What the Pipeline Does Not Do

The integration pipeline does not guarantee full test coverage by itself.

It also does not try to replace:

- developer review
- fixture design
- assertion design
- domain knowledge
- runtime validation
- coverage measurement
- mutation testing
- normal test maintenance

It can identify seams and provide structured context, but a test still needs
real setup, real assertions, and a meaningful definition of correctness.

The pipeline also does not attempt to model every possible runtime path. It
works from representative execution items and analyzed integration paths. That
is deliberate. The goal is useful testing guidance, not exhaustive execution
simulation.

## Relationship to Unit Analysis

Integration analysis depends on unit analysis.

The unit pipeline owns source-level decomposition. It identifies structural
entries, bindings, callables, execution items, integration candidates, and
paths to integration points inside a unit. The integration pipeline should not
redo that work from scratch.

Instead, integration analysis consumes those unit inventories and derives
project-level relationships from them.

In practical terms:

1. Run unit analysis first.
2. Run integration analysis against the same target project.
3. Use the generated specs for review or agent-assisted test generation.

The root CLI can run both steps:

```bash
pybastion all /path/to/project \
  --readiness \
  --check-graph \
  -v
```

The integration pipeline can also be run directly after inventory exists:

```bash
pybastion integration \
  --target-root /path/to/project \
  --check-graph \
  -v
```

## Validation Expectations

The integration pipeline should preserve checkable invariants.

Examples include:

- every graph node should correspond to known inventory information
- every source execution item should belong to its containing callable
- every source callable should belong to its containing unit
- every resolved target unit should exist in the inventory set
- every integration spec should refer to a valid source unit
- every split spec should preserve the source and target relationship
- graph checker findings should be visible rather than silently repaired
- unresolved or uncertain targets should remain visible as uncertainty

Validation is especially important because the output may be used by an agent.
If the analysis is uncertain, stale, or incomplete, the artifact should say so.
A quiet guess is worse than an honest gap.
